# Messaging redesign — релей, надёжная доставка, синк истории, шифрование

Дизайн-документ (не реализация). Фиксирует обсуждённую целевую модель для
сообщений/каналов/заметок поверх шины. Текущий код описан в `ARCHITECTURE.md`;
этот файл — про то, **куда мы хотим прийти и почему**.

## 1. Что не так с текущей моделью

Сейчас (см. `services/messenger`, `mesh/bus`):

- **Доставка** — автор фанаутит N копий каждому участнику напрямую
  (`emitTo` → `go deliver` на каждого), редоставка через свой SQLite-outbox
  (at-least-once, ретрай раз в 30с, TTL 14 дней). Сам механизм очереди
  **исправен** (проверено тестом), но:
- **Только автор ретраит своё, пиры чужое не релеят** (no-signature trust:
  личность = оверлейный IP, подтверждённый AWG). Следствие: пир за симметричным
  NAT (недостижим напрямую, но достижим через соседа) — **не получит** ничего.
- **Нет бэкафилла / anti-entropy.** Если получатель был офлайн, а автор потом
  ушёл/потерял outbox — сообщение **потеряно навсегда**. Достать его у того, кто
  получил, нечем. Истории расходятся и **никогда не сходятся**.
- **Новый участник канала видит пустую историю** — ему льют только с момента
  добавления, бэклог никто не отдаёт.
- **Шифрования нет вообще** — payload едет plaintext-JSON. Шифрует только AWG
  hop-by-hop; на каждом пире payload расшифрован. e2e нет → будущий релей будет
  читать всё.
- **Узкое место надёжности**: сообщение попадает в outbox только ПОСЛЕ падения
  асинхронного `deliver` (≤10с) — краш в это окно теряет доставку.

## 2. Целевая модель: реплицируемые подписанные логи + anti-entropy

Ключевая идея: вместо «толкаем сообщение» — **у каждого автора в каждом scope
свой append-only подписанный лог, пиры его реплицируют**. Это модель
Scuttlebutt/Briar. Одним махом закрывает **доставку, редоставку, релей,
мультикаст и синк истории** — это всё свойства репликации, а не четыре
отдельных механизма.

- `scope` = id канала (`<owner>/<name>`) или pub собеседника (DM).
- Лог ключуется по **`(author, scope)`** (НЕ просто по автору): «история канала
  X» = объединение под-логов `(*, X)`. Так синк тянет ровно нужный срез, без
  чужих DM/каналов и без утечки.
- **Релей/мультикаст** = «любой пир отдаёт записи любого автора, что у него
  есть». Автор офлайн — неважно, бэкафиллит кто угодно.
- **Редоставка** = автоматом через репликацию (не нужен author-only-outbox).
- **Синк истории / новый пир** = обмен **фронтиром** `{author → seq}` по scope с
  пустого старта.
- **Дедуп/целостность** — по `(author, scope, seq)` + хэш-цепочка `prev`.

Это поглощает ad-hoc outbox + фанаут. Архитектурно крупнее (сдвиг от
push-мессенджинга к **репликации состояния / CRDT-ish**), но радикально надёжнее.

## 3. Конверт: внутренний (e2e) + внешний (маршрутный)

Двухслойный, как в Briar/libp2p-relay.

**INNER (`Entry`)** — неизменяемая, подписанная автором запись лога. Релей не
может подделать/(если шифр) прочитать:

```go
type Entry struct {
    Author  string  // ed25519 pub
    Scope   string  // id канала / pub DM
    Seq     uint64  // монотонно в (Author, Scope)
    Prev    string  // хэш предыдущей записи → цепочка (порядок + tamper-evidence)
    Epoch   uint64  // какой group-key нужен (см. §6)
    TS      int64
    Type    string  // text / add / remove / keywrap / notes / call_* / ...
    Body    []byte  // ciphertext = secretbox(epochKey, plaintext)  (или plaintext в Ф0)
    Sig     []byte  // ed25519(Author) по канону всего выше
}
```

**OUTER (`Relay`)** — маршрутный слой, переписывается на каждом хопе:

```go
type Relay struct {
    Forward []string // подмножество получателей, за которое отвечает ЭТОТ хоп
    TTL     uint8    // hop-limit от петель
    Entry   Entry    // вложенная неизменяемая запись
}
```

Релей: проверил `Entry.Sig` → для каждого в `Forward` кого достаёт сам доставляет,
кого нет → передаёт следующему релею (переписав `Forward`, уменьшив `TTL`) или
кладёт в store-and-forward. Получатель: проверил подпись → дедуп по
`(Author,Scope,Seq)` → **e2e-ACK по id** обратно автору. Внутренний слой подписан
и неизменен — отсюда безопасность релея.

## 4. Релей

- Концептуально это **транспорт/маршрутизация** → место в шине, не в мессенджере.
- Ломает текущую модель «личность = оверлейный IP» (она подтверждает только
  ПОСЛЕДНИЙ хоп). Поэтому релеящиеся записи **обязаны быть подписаны автором** и
  проверяться e2e. Для приватности от релея — ещё и зашифрованы (см. §6).
- **Выбор релея**: общий достижимый сосед. Кого видно — берём из gossip
  (peer-view уже реплицируется). Для NAT-кейса достаточно single-hop.
- Защита от петель/усиления: `TTL`/hop-limit + дедуп по всей сети.

## 5. Мультикаст (доставка многим)

- `bus.Deliver(toPubs []string, typ, payload)` — конверт адресуется **множеству**.
- Прямо — кому автор дотягивается; недостижимых отдаём релею подмножеством, он
  фанаутит/очередит уже его → 1-уровневое дерево (автор → пара релеев → листья).
  Снимает O(N) с автора и закрывает NAT.
- Дедуп `(Author,Scope,Seq)` гасит задвоение от нескольких путей.
- Канальный фанаут мессенджера схлопывается в один `bus.Deliver(members, ...)`.

## 6. Динамическое членство + история «только с момента вступления»

**Ростер — не отдельное мутабельное состояние, а факт в логе.** Изменения состава
(`add`/`remove`) — записи в scope-логе, **авторизованные овнером** (он
единственный авторитет, как и сейчас). Текущий ростер = **свёртка** owner-записей.
Single-writer → тотальный порядок → без CRDT-конфликтов. Хак «таскать снимок
`Members` в каждом сообщении» уходит.

**Решение: новый участник видит историю ТОЛЬКО с момента вступления.** Крипто-чисто
реализуется эпохами и group-key (граница вступления = смена ключа, а не «политика
выдачи»):

- У канала монотонный **epoch**; овнер бампает его на **каждое** изменение состава.
- У каждой эпохи свой симметричный **group key**. Сообщения эпохи K шифруются
  ключом K. Вступивший в эпоху K имеет ключи K, K+1, … но **не** K−1 → прошлое
  прочитать нечем. Удалённый не получает K+1 → будущее закрыто.
- **Раздача ключа**: на смене эпохи овнер заворачивает новый key каждому текущему
  участнику на его **x25519-pub** (`nacl/box`). Это записи `Type=keywrap` в том же
  логе (`KeyWrap{epoch, forMember, box(key)}`), упорядоченные и tamper-evident.
  Стоимость O(N) **на изменение состава** (редко), не на сообщение.
- Анти-энтропия гоняет **ciphertext** свободно → релей/бэкафилл безопасны
  by-construction; `IsMember` остаётся лишь оптимизацией (не лить чужим зря), а не
  гарантией.
- Удаление = новая эпоха + ротация; старое у ушедшего остаётся (append-only,
  «развидеть» нельзя), будущее закрыто.
- Овнер ушёл навсегда → состав замораживается (owner-keyed id). Делегирование
  админки — потом.

## 7. Шифрование: состояние и план

**Сейчас:** прикладного шифрования НЕТ (ни каналы, ни DM) — plaintext-JSON, шифрует
только AWG hop-by-hop. e2e отсутствует.

**Почему группа — тяжёлый случай:** 1:1 тривиально (`box` на x25519 собеседника).
Боль группы: (1) раздача ключа N участникам, (2) rekey при смене состава, (3)
forward secrecy / post-compromise — ради неё существуют **MLS (RFC 9420, TreeKEM)**
и Signal Sender Keys + ratchet.

**Наш epoch-дизайн** (§6) закрывает конфиденциальность + границы from-join/from-leave
без тяжёлого TreeKEM. Честные минусы v1:
- **Нет forward secrecy внутри эпохи** — один ключ на эпоху; утёк → вскрыта вся
  эпоха (между эпохами отсечено ротацией).
- **Овнер — центр доверия по ключам** (может слить epoch key не-участнику). Полной
  децентрализации (вклад каждого, как в MLS) нет.

Эпохи уже заложены в формат → до MLS-грейда (FS/PCS) можно дорасти, не ломая wire.

**Ключи.** identity уже даёт обе примитива: `ed25519` (подпись) и производный
`x25519` (ECDH, сейчас это WG-static). Для messaging лучше вывести **отдельный**
x25519 (другой HKDF-info), чтобы не смешивать с WG-static.

## 8. Библиотеки — решение

**Без фреймворка.** Тонкий слой репликации пишем сами поверх нашей шины.
Зависимости — только уже имеющееся:
- подпись — `crypto/ed25519` (стдлиб);
- шифр — `golang.org/x/crypto/nacl/box` (1:1 / keywrap) + `nacl/secretbox` (group key);
- хранилище — наш SQLite (`modernc.org/sqlite`);
- синк — наш код (с per-author `seq` он простой).

**Отвергнуто:** `go-libp2p` (gossipsub + circuit-relay v2 — буквально оно, но это
целый сетевой стек, заменяющий наш AWG+QUIC-bus → нет); `NATS` (серверный); MLS/
Signal-ratchet (тяжело, рано). `go-ssb`, **Negentropy** (range-based set
reconciliation из nostr) — **только как референс дизайна, не зависимость**. При
per-author `seq` синк тривиален (обмен фронтиром), фэнси set-recon не нужен.

## 9. Зафиксированные решения

- Целевая архитектура: **реплицируемые подписанные `(author, scope)`-логи +
  anti-entropy**, поглощающие доставку/редоставку/релей/мультикаст/синк.
- Конверт **двухслойный** (inner подписан/неизменен; outer маршрутный/мутабельный).
- Релей — на уровне **bus**, single-hop для старта, недоверенный (e2e-подпись).
- Членство — **owner-authored записи в логе**, ростер = их свёртка.
- История новому участнику — **только с момента вступления** (эпохи + group key).
- Шифрование v1 — **epoch group key** (без intra-epoch FS); DM — `box`.
- Без сторонних фреймворков; стдлиб + `x/crypto/nacl` + SQLite.

## 10. Открытые вопросы

1. **Шифр релеящегося в v1**: вообще без (подпись + членский доступ) или сразу
   epoch group key? (От этого зависит объём первой фазы.)
2. **Forward secrecy внутри эпохи** — нет в v1; когда добавлять ratchet/MLS?
3. **Кросс-авторский порядок**: `TS` (как SSB) или Lamport-часы?
4. **Обрезка логов** — хранить всё / лимит по времени-объёму?
5. **Кто может быть релеем** — любой общий достижимый сосед или назначенные
   «релей-узлы» (белый IP/VPS)?

## 11. Возможное фазирование

- **Ф0** — подписанный `Entry` + перенос reliable-deliver (outbox) с messenger в
  bus, **прямая** доставка (без релея). Чистит архитектуру, готовит почву. Заодно
  чинит «узкое 10с-окно» (enqueue-on-send).
- **DM e2e** (`box`) — маленький изолированный шаг, польза сразу, можно до всей
  лог-репликации.
- **Ф1** — single-hop релей (закрывает NAT, разгружает фанаут) + anti-entropy синк
  (фронтир `{author→seq}` по scope + запрос диапазонов + проверка цепочки).
- **Ф2** — epoch group key для каналов (конфиденциальность + границы from-join/leave).
- **Ф3** — multi-hop релей по gossip, ratchet/MLS (FS/PCS) — если понадобится.
