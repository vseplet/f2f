# Хранилище секретов

> **Статус:** базовое хранилище реализовано (`services/secrets`, вкладка
> **secrets**); шифрование/tiers/unlock — дизайн (ниже).

Хранилище секретов для f2f: инфра-секреты узла, секреты поднимаемых сервисов и
личные секреты пира. Опирается на identity (`peer_pub`, см.
[IDENTITY.md](IDENTITY.md)); доступ гейтится членством в канале
(`channels.IsMember`).

## Реализовано

Doppler-подобная модель **vault → environment → secret** (`services/secrets`,
`/api/secrets*`, вкладка **secrets**):

- **Vault** — проект (контейнер), сам по себе не шарится.
- **Environment** — конфиг внутри vault'а (`dev`/`staging`/`prod`…). Несёт
  **свои** key/value **и свой список шаринга** — поэтому `prod` можно отдать в
  один канал, а `dev` — в другой, независимо. Окружения **плоские** (без
  наследования) — доступ предсказуем.
- **Secret** — `{key, value}` внутри окружения.

**Хранилище — НЕ блок-лог, а отдельный per-camp sqlite** (`<camp>/secrets.db`):
секреты должны быть **мутабельными и по-настоящему удаляемыми**, а append-only
лог хранил бы каждое старое значение в открытом виде навечно и реплицировал бы
его неудаляемо. Поэтому обычные `UPDATE`/`DELETE`.

**Шаринг — «owner-served pull»** (как у drop): каждый пир хранит свои vault'ы;
окружение, расшаренное в канал, **не реплицируется**, а отдаётся членам этого
канала **по запросу** через bus (`secrets.list`), гейтясь `IsMember(bid,
fromPub)`. Получатель видит чужие окружения **живьём** и **read-only** (помечены
`mine=false`, `owner`/`owner_name`); владелец правит только своё. Цель шаринга —
**канал ИЛИ DM** (`/api/secrets/targets` отдаёт и каналы, и личные, с лейблом
собеседника); шар в DM = оба пира видят окружение.

**Имена каналов резолвятся локально** из членства (scope `channel:<bid>`
синкается только членам). Поэтому чужое окружение показывает только те каналы, в
которых состоит **смотрящий** (фильтр `via` на фронте) — bid'ы каналов, куда его
не звали, не утекают.

**Пока НЕ делает** (см. дизайн ниже): шифрование/tiers, keywrap по членам,
passkey/PRF-unlock, ротацию при revoke. **Значения лежат в открытом виде** в
sqlite (по зашифрованному транспорту, но at-rest cleartext).

## Что у нас уже за секреты

| секрет | где сейчас | класс |
|---|---|---|
| identity `priv.key`, CA-key, `oidc_rsa.pem` | `/var/lib/f2f/…`, камп-дир, root `0600` | инфра узла |
| OIDC `client_secret` | `clients.json` (**plaintext**) | сервис-секрет |
| Gitea `SECRET_KEY`/`INTERNAL_TOKEN`/`JWT_SECRET`, БД-пароли | конфиги сервисов | сервис-секрет |
| API-ключи, пароли, токены пользователя (будущее) | — | личный |

## Три класса — разные требования

1. **Инфра узла** — нужны процессу при старте, без человека (иначе узел не
   поднимется). Шифрование к device-ключу максимум; «запаролить» нельзя
   (headless). Скорее остаются как есть, опционально wrap к device-x25519.
2. **Сервис-секреты** — нужны сервису автоматически (DB-пароль на boot).
   Без человека, но хочется **синка между членами канала** (кто совместно
   держит сервис).
3. **Личные секреты** — «запароленное / passkey / с диска просто так не
   читается».

## Ключевое противоречие

- **Синк + headless** требуют, чтобы ключ был **на диске** (wrap к
  device-x25519), иначе вторая машина / сервер не расшифруют сами.
- **«Запароленное»** требует, чтобы ключ **НЕ** лежал на диске целиком
  (иначе root читает без человека).

В одном ведре несовместимо → **два уровня защиты**:

- **operational** (классы 1–2): ключ wrap'ается к **device-x25519**,
  расшифровка без человека, синкается как есть. Это то, что сервисам/узлу
  нужно автоматически.
- **sealed** (класс 3): gated **passkey/паролем**, с диска в одиночку не
  открыть, между машинами синкается **только шифротекст**, раскрытие —
  интерактивно на устройстве с человеком.

## Синхронизация

Модель пир=пользователь убирает «синк между моими устройствами» (у пира одно
устройство). Остаётся два случая:

- **Личный секрет** — принадлежит пиру, **не синкается**: лежит локально, wrap к
  его собственному ключу (operational) / sealed. Доступен только этому пиру.
- **Секрет сервиса/проекта** (scope = channel) — синкается **членам канала**:
  симметричный ключ хранилища **wrap'ается к x25519 каждого члена канала**
  (keywrap из MESSAGING_DESIGN.md). **Добавили члена** → wrap к нему; **убрали
  члена** → **ротация** ключа + re-wrap на оставшихся (исключённый не откроет
  новые записи).

Никакой новой крипты — переиспользуем x25519 (выводится из camp-ключа) и
keywrap-схему. Состав канала — из `block.channel.members`.

> **Recovery стало острее.** Раньше потерю машины спасало второе устройство того
> же юзера; теперь машина пира = единственная копия его личных секретов. Нужен
> явный recovery (recovery-код / пароль / экспорт), иначе потеря машины =
> безвозвратная потеря личных секретов. См. открытые вопросы.

## Unlock для sealed — ОТКРЫТО (решить)

Важно: **passkey сам по себе не даёт ключ шифрования** (WebAuthn — это
аутентификация). Варианты master-механизма:

| вариант | как | плюс / минус |
|---|---|---|
| **Passkey-PRF** (hmac-secret) | Touch ID → PRF выводит симметричный ключ (как Bitwarden) | без пароля, в духе наших passkey; **нужен PRF-support** (свежие Apple ок, security-keys как повезёт) |
| **Пароль (Argon2id)** | master-пароль → KDF → ключ | работает везде, просто; «ещё один пароль» |
| **PRF + пароль-fallback** | PRF где есть, пароль для recovery / устройств без PRF | гибко и надёжно; больше кода |

Рекомендация — **PRF + пароль-fallback** (PRF как основной путь в духе f2f,
пароль как recovery и для headless-восстановления). Но решение **отложено**.

## Модель доступа

- Секрет привязан к scope: **peer** (личный), **channel** (секрет сервиса/
  проекта, виден членам канала), **camp/node** (инфра).
- Чтение personal-секрета — только владелец-`peer_pub` (после unlock).
  Channel-секрет операционно доступен членам канала / их сервисам
  (operational-уровень).
- Будущие ACL/группы (см. ARCHITECTURE.md / OIDC allowlist) могут расширять:
  «секрет X виден группе @ops».

## Формат (набросок)

- Хранилище — per-camp (рядом с `clients.json`/`passkeys.json`): JSON/db с
  записями `{id, scope, name, tier, ciphertext, meta}`.
- Каждая запись зашифрована ключом своего tier'а; ключи tier'а wrap'нуты к
  peer-x25519 (operational) и/или к passkey-PRF/паролю (sealed). Channel-секреты
  — wrap к x25519 каждого члена канала.
- Channel-секреты реплицируются синком лога (scope канала); peer-секреты —
  локальны, не реплицируются.

## Что меняется в коде (наметки)

- `clients.json` `client_secret` → перенести в хранилище (сейчас plaintext).

## Открытые вопросы

- **Unlock-механизм** для sealed (см. выше) — главное нерешённое.
- **Recovery личных секретов при потере машины пира** — острее без мульти-девайс
  копии: recovery-код? пароль? экспорт? Без этого потеря машины = потеря личных
  секретов навсегда.
- Нужен ли sealed-уровень headless-узлам вообще (вероятно нет — только
  operational).
- Ротация инфра-секретов (oidc_rsa, CA) — отдельная тема.

## Поэтапно

1. **operational-уровень** + `services/secrets` + перенос OIDC `client_secret`
   из plaintext в хранилище (быстрый и полезный первый шаг, без человека).
2. **Синк channel-секретов** по членам канала (keywrap + ротация при revoke).
3. **sealed-уровень** + выбранный unlock (passkey-PRF/пароль) + UI.
4. **Revoke/ротация**, recovery, ACL по группам.
