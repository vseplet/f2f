# Блоки: распределённый движок документов, задач и чата

> **Статус:** ядро реализовано (`db/blocks`); на нём работают **каналы,
> сообщения и заметки (со страницами)**. Tabs/merge UX, GC/компакция, e2e —
> дизайн/частично.

Единая блок-модель поверх реплицируемого лога (`db`): **чат, документы и
таск-трекер — это одно приложение из блоков**, а не три системы. user-identity —
[INVITE.md](INVITE.md).

## Реализовано

- **`db/blocks`** — движок: `Manager` над `db.Service`, операции `Create`/
  `Update`/`UpdateType`/`Move`/`Delete`/`Merge`/`Upsert`, свёртка `Blocks(scope)`
  → `Block{BID,Type,Channel,Parent,Pos,Heads,History,Deleted}`. **Инкрементальная
  свёртка** (кэш на scope, доворачивает только дельту). bid namespace'ится
  автором: `<fp16>-<rand>`; `Upsert` принимает заданный bid (well-known/derived).
- **`db/blocks/channels`** — `block.channel` `{name, members}` в своём scope
  `channel:<bid>` (per-channel — чтобы синк гейтил по членству);
  `general` (well-known bid), DM (`dm-<hash(pubs)>`), `IsMember`.
- **`db/blocks/message`** — `block.message` `{body,file,reply_to,thread}` в scope
  `message:<channelBid>`; правка = новая версия (история бесплатно).
- **`db/blocks/attach`** — общий примитив вложения `Attachment`
  `{name,mime,size,data?|info_hash?+magnet?}`: мелкие инлайном (≤8 МиБ), крупные
  по torrent (drop). Переиспользуют и сообщения (поле), и заметки (блок).
- **Заметки** — generic `block.text` `{md}`, `block.page` `{title}` и
  `block.file` `{…attachment, caption?}` в scope `note:<conv>`; вложенность
  страниц — через `Parent`. Вложение = самостоятельный `block.file` среди
  текстовых (Notion-стиль), а не поле текстового блока.
- **Web-API per-entity**: `/api/channels`, `/api/messages`, `/api/notes`,
  единый push-стрим `/api/events`; generic `/api/blocks` — внутренний.

## Идея

Распределённая БД, где **коллизии невозможны by construction**: ничего не
редактируем и не удаляем — только **накапливаем иммутабельные подписанные
события**. Состояние = детерминированная **свёртка (fold)** над историей.
Это модель git (content-addressed объекты + DAG) и event sourcing.

Блок — **универсальный атом**:
- сообщение чата = append-блок,
- задача = блок с полями,
- абзац/заголовок/тудушка документа = блок.

Отличается только тип блока и его поля. Notion/AnyType устроены так же
(AnyType — живой p2p/local-first пример).

## Иммутабельность

- **Нет update/delete.** «Редактирование» = новое событие-версия.
  «Удаление» = событие-тумбстоун (тоже **добавление**, физически ничего не
  стирается → аудит и откат возможны).
- Truth = лог событий; состояние блока = свёртка над его историей.
- События подписаны автором (`user_id`), реплицируются и не мутируют.

## Блок = хэш-DAG версий

Каждое событие-версия ссылается на **родительскую версию(и)** (хэш = causal
context). Получается DAG:

```
      v1
     /  \
   v2а   v2б     ← правили параллельно: обе ветки живут (siblings)
     \  /
      v3         ← merge-событие: 2 родителя, выбранный/совмещённый контент
```

- **heads** DAG'а = текущее значение блока. Один head → разрешено;
  несколько → нерешённые варианты.
- Линейная цепочка (потомок предка) → supersede, конфликта нет.

## Порядок — логический, НЕ wall-clock

В распределёнке нет надёжных глобальных часов. Порядок:

1. **Каузальный (из DAG):** A предок B ⇒ A < B, гарантированно.
2. **Конкурентные** (ни одно не предок другого) каузального порядка не имеют —
   это **ровно siblings**. Для детерминированного тотального порядка
   (дефолт-выбор/рендер) — **Lamport-счётчик + tie-break по author/hash**.
3. **Wall-clock** — только метадата для показа человеку, не источник истины.

## Конфликты: siblings + явный merge, БЕЗ авто-резолва

Система **никогда сама не пишет резолюцию**. Конкурентные правки = варианты,
которые сосуществуют:

- **Нерешённый блок (heads > 1) рендерится как табы**: «вариант 1 — Alice»,
  «вариант 2 — Bob». Не блокирует, ничего не теряет.
- Ярлык таба — **автор + время (+ короткий хэш)**; номер **локальный**
  («вариант 1/2/3»), не идентичность (в DAG глобального «v2» нет — у разных
  пиров разъедется).
- **Дефолтный таб** детерминированный (последний по lamport + tie-break) —
  чтобы рендер/экспорт/запросы стабильно брали один.
- **Резолв = выбрать таб** → подписанное `block.merge { parents:[все heads],
  content }` → схлопывается в один head. Резолв **необязателен** — можно жить
  табами сколько угодно. Merge несёт provenance (кто слил) и откатывается ещё
  одним событием.
- **Само-конкуренция**: свои два устройства просто дают два таба (разобрать
  тривиально). Опционально — лёгкая линеаризация правок одного `user_id` (один
  человек — не конфликт, берём последний), чтобы свои мелкие правки даже табов
  не плодили. Это опция, не необходимость.

## Документ как набор блоков

Документ = три простых CRDT-примитива (тяжёлый char-level CRDT над всем
текстом **не нужен** — атомарность блоков его убирает):

| что | CRDT | сложность |
|---|---|---|
| множество блоков (по `block_id`) | OR-Set (add + тумбстоун) | легко |
| порядок блоков | позиции = fractional index, tie-break по author/id | легко |
| контент блока | DAG версий + heads (см. выше) | средне |

`block_id` глобально уникален (случайный) → блоки не сталкиваются. Позиции
fractional + tie-break → конкурентная вставка «между A и B» сходится одинаково
у всех.

## События (намётка)

```
block.create { id, type, pos, content, parents:[] }
block.update { id, content, parents:[<head hashes>] }   // новая версия
block.move   { id, pos }
block.delete { id }                                      // tombstone
block.merge  { id, parents:[<head hashes>], content }    // явный резолв
```

Все подписаны `user_id`, добавляются в per-author лог соответствующего scope.

## Схема записи: `Frame` ⊃ `op`

Блок живёт как payload внутри `db.Frame` (см. [DB.md](DB.md)). Раздел
cleartext/sealed критичен: метадата открыта (поиск/релей), контент запечатан
(шифрование — дизайн; сейчас payload открыт).

```
Frame {                       ← cleartext (db индексирует, релей носит вслепую)
  scope, author, seq, prev, lamport,
  type     "block.text"|"block.channel"|"block.message"|"block.page"…  ← индекс
  ts, sig, id
  payload  <op>               ← сейчас открыт; e2e-seal — дизайн
}

op {                          ← внутри payload, app-уровень
  bid      string             // стабильный id атома (random) — один на ВСЕ версии
  op       create|update|move|delete|merge
  parents  []entryID          // version-DAG: версии, на которых основан / которые мёржит
  pos      string             // fractional index в контейнере (порядок/move)
  parent   string             // bid контейнера (вложенность); "" = корень
  content  {…}                // по типу; нет у delete
}
```

Два разных «предка»:
- `Frame.prev` — предыдущая запись **этого автора** в логе (per-author chain) →
  целостность репликации.
- `op.parents` — версии **одного блока** (cross-author) → heads/табы/merge.

`content` по типам (✅ реализованы / 📐 дизайн):
```jsonc
block.text     { "md": "привет **мир**" }                        // ✅ заметки/доки
block.page     { "title": "Design" }                             // ✅ страница-контейнер; дети по Parent
block.file     { "name":"spec.pdf", "mime":"…", "size":N, "data"|"info_hash"+"magnet", "caption"? }  // ✅ вложение-блок
block.channel  { "name": "devs", "members": ["<pub>", …] }       // ✅ канал
block.message  { "body": "ало", "file": {…}, "reply_to": "<bid>", "thread": "<bid>" }  // ✅ сообщение
block.todo     { "text": "купить молоко", "done": false }        // 📐
block.table    { "cols": [...], "rows": [[...]] }                // 📐
```

**page vs text:** `text` — это контент (несёт markdown). `page` — контейнер:
несёт только `title`, его «тело» = блоки с `Parent == bid` страницы. Вложенность
страниц рекурсивна (любая глубина). Notion-модель.

Поиск: по `Entry.type` — быстрый метадата-фильтр без расшифровки; полнотекст по
`content` — на стороне приложения после decrypt (проекция с FTS).

## Ложится на существующие рельсы

- **scope** (документ / список задач / канал) = namespace в логе.
- **репликация, anti-entropy, синк между устройствами** = MESSAGING_DESIGN.
- **identity / membership / ACL / e2e (group keys)** = как у каналов; новый
  «app» = новый тип блока, остальное бесплатно.

Три приложения — проекции над одним логом:
- **чат** — append-блоки, порядок по causal/lamport (тривиально);
- **таски** — блоки с полями; конкурентный статус = два head'а = табы (для
  таск-трекера это даже желательно — видно, что двое разошлись);
- **доки** — упорядоченные блоки. Если когда-то нужен символьный мёрж **внутри**
  одного блока — точечно вешается крошечный Yjs только на тот блок; по умолчанию
  блок атомарен (variant-табы).

## GC / компакция

Append-only растёт бесконечно → нужна **компакция**: после того как вариант
разрешён (merge) и реплика разъехалась всем, перекрытые версии можно дропнуть.
Это оптимизация хранения; **модель остаётся иммутабельной** (truth = лог,
состояние = свёртка).

## Референсы

- **AnyType** — p2p/local-first Notion на блоках (пруф, что подход рабочий).
- **git** — иммутабельные объекты, DAG, merge-коммит с несколькими родителями.
- **Dynamo/Riak** — siblings (multi-value register) вместо перезаписи.
- **Automerge/Yjs** — зрелые CRDT, Yjs для опционального текста-внутри-блока.

## Открытые вопросы

- **Вложенные блоки** (поддеревья, как в Notion) — variant-табы над поддеревом
  vs только над листовым контентом (вероятно начать с листьев).
- Схема **fractional index** (LSEQ / dense order) и её tie-break.
- Линеаризовать ли правки одного `user_id` (само-конкуренция) — см. выше.
- **GC-политика**: когда безопасно дропать перекрытые версии (нужен признак
  «все реплики видели»).
- e2e для блок-событий (group keys per scope) — наследуется из messaging.
