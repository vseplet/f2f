# Идентичность, членство и аутентификация — на блоках

> **Статус:** 📐 дизайн. Кладёт на блок-движок (`db/blocks`) три вещи: **кто ты**
> (профиль пира), **пускают ли тебя в кэмп** (членство, контролирует владелец),
> **чем ты входишь** (passkey/OIDC). Связано: [INVITE.md](INVITE.md) (инвайты/
> допуск), [OIDC.md](OIDC.md) (провайдер — реализован), [SECRETS.md](SECRETS.md)
> (vault), [ENTITIES.md](ENTITIES.md) (модель сущностей).

## Идея

Личность, членство и аутентификация — это всё **подписанные иммутабельные факты**,
ровно то, для чего есть лог `db`. Кладём их блоками. Одно **жёсткое правило**:

> **Публичное — в блоки. Секреты — нет.** payload блока сейчас открыт (e2e-seal —
> дизайн), поэтому любой приватный ключ в блоке утечёт во все реплики.

> **Модель: пир = пользователь.** Отдельного слоя `User`/`user_id` нет.
> Идентичность = **`peer_pub`** (ed25519 машины). У пира есть **профиль** —
> отображаемая часть (имя, first/last, email, публичные passkey-креды). Несколько
> устройств = несколько пиров. См. [ENTITIES.md](ENTITIES.md).

## Слой 1 — Профиль: `block.profile` (self-authored)

- **scope** well-known `profiles` — **один общий скоуп на все профили кэмпа**,
  синкается всем членам (как `camp`). Не per-peer-скоуп.
- **ключ блока** = `peer_pub`; **автор = сам пир**.
- **Правило валидности (ключевое):** `block.profile` для пира X принимается
  **только если подписан pub'ом X**. Подделать чужой профиль нельзя. Один автор
  на профиль ⇒ конфликтов/siblings нет, хотя скоуп многописательный (как
  `message:<bid>`: много авторов в скоупе, но каждый пишет только своё).
- **content:**
  ```jsonc
  block.profile {
    name,                      // = имя пира (username), уникально в кэмпе
    first, last,
    email,                     // <name>@<zone>.f2f — для OIDC
    passkeys: [<public webauthn credential>, …]  // ТОЛЬКО публичные креды
  }
  ```
- Приватный ключ passkey не наш — он не покидает аутентификатор, хранить нечего.
- **Зачем блоком:** профиль синкается всем → публичная часть passkey есть у всех
  → любой co-located IdP проверяет ассерцию гостя по креде из лога. Это закрывает
  TODO в `services/oidc/credstore.go` («camp-wide replication of the public
  credential») — синк профиля **и есть** репликация кред.

## Слой 2 — Членство в кэмпе: `block.camp` (owner-only)

То, что раньше называли «список пиров». Это **верхний ACL кэмпа**.

- **scope** well-known `camp`; **single-writer = владелец** (pub из `camp_id`
  `<owner_pub>_<label>`).
- **Правило валидности (ключевое):** версия `block.camp` принимается **только если
  подписана владельцем**; любые чужие версии фолд отбрасывает. «Контролирует только
  владелец» получается **by construction — через подпись**, а не через проверку на
  сервере. Single-writer ⇒ ни конфликтов, ни siblings/tabs.
- **content:**
  ```jsonc
  block.camp {
    name,                          // имя кэмпа
    members: ["<peer_pub>", …],    // кто допущен в кэмп
    admins?:  ["<peer_pub>", …]    // опционально: кому делегировано право менять состав
  }
  ```
- **Член кэмпа ⟺ его `peer_pub` в этом блоке.** Camp-сервер больше **не решает**,
  кто внутри — он становится чистым рандеву/релеем (можно считать недоверенным).

## Слой 3 — Аутентификация / OIDC

- **`block.oidcclient`** (scope `channel:<bid>` или `camp`): публичный конфиг клиента
  ```jsonc
  block.oidcclient { client_id, name, redirect_uris, logout_uris, confidential }
  ```
  allowlist «кому можно входить» = **членство канала** (участники). Выигрывает от
  синка между пирами и от истории версий (аудит).
- **Публичный JWK** — может быть блоком, но и так раздаётся на well-known эндпоинте.
- **НЕ в блоки (секреты):**
  - **client secret** (сейчас в `clients.json` плейнтекстом) и
  - **приватный RSA-ключ IdP** (`signkey.go`)
  → в **vault** ([SECRETS.md](SECRETS.md), device-wrapped/sealed), либо
  **per-peer-локально**. Реплика приватного ключа IdP = любой пир сможет подписывать
  токены как этот IdP → ломается per-peer attestation. **Подпись остаётся per-peer.**
- **Проверка vs подпись:** проверить passkey-ассерцию может **любой** IdP (общая
  публичная креда из `profiles` + общий `rp.id=<zone>.f2f`); **подписать** токен —
  только тот IdP, что реально провёл вход (его per-peer RSA-ключ). См. [OIDC.md](OIDC.md).

## Как подключается к движку

| что | сейчас | становится |
|---|---|---|
| членство кэмпа | camp-announce (кто заанонсился) | `block.camp.members` (owner-signed) |
| `Sync.SetMemberCheck` | гейт по членству **канала** | + верхний гейт: не в составе кэмпа → ничего не реплицируем |
| `handlePairReq` (допуск пиров) | дроп не-членов по camp-роли | источник членства — `block.camp` |
| camp-announce | допуск + liveness | только **liveness** (кто сейчас online) |
| членство канала | `block.channel.members` | без изменений — **нижний слой ACL** |

Два слоя: `block.camp` (кто в кэмпе, owner) ⊃ `block.channel.members` (кто в канале,
владелец канала).

## Где что хранится — три слоя

Это карта хранения для всего identity/membership/ресурсов. Три слоя по природе
данных:

| слой | что | свойства |
|---|---|---|
| **DB** (блоки) | профили (`block.profile`, scope `profiles`), членство кэмпа/каналов (`block.camp`/`block.channel`), инвайты-мета (`block.invite`), OIDC-клиент (публичный конфиг), messages/notes/files-ref | реплицируется, подписано, **единый ACL-источник** |
| **config** (локально) | ключ пира (ed25519); identity пира (имя, listen-порт, WG); **публикуемые ресурсы** (domains/files/intercepts/egress/shell/VNC); **полиси-привязки** ресурс→группа; секреты (client secret, приватный ключ IdP — лучше vault) | durable, per-peer, **не реплицируется** |
| **memory** (рантайм) | liveness (online/offline), текущие endpoint'ы, RTT, pair-state, кэш announce | волатильно, пересобирается из camp + DB |

Правило: **durable shared → DB; durable local/secret → config; volatile/derived → memory.**

### Полиси ∘ членство (как композируется доступ)

- **Полиси-привязка** (`my-domain → channel:devs`) — **config**: локальное решение
  пира про свой ресурс.
- **Состав группы** (`кто в channel:devs`, `кто в кэмпе`) — **DB-блок**
  (`block.channel.members` / `block.camp`, подписан владельцем группы).
- Проверка доступа = **config (привязка ресурс→группа) ∘ DB (членство группы)**.
  Так ресурсы и их политика — локальная власть пира, а идентичность
  принципалов — общая правда; **копии членства не разъезжаются** (единый источник).

> **Почему членство нельзя в config:** если каждый пир держит свой список
> «кто в кэмпе/канале», копии разойдутся и единого ACL не будет. Членство — всегда
> один подписанный блок в DB; config лишь **ссылается** на группу.

### Что выпиливается

`<camp_id>.config.json` сейчас держит **`PeerCatalog`** (персистнутый список пиров).
С новой моделью он **не нужен**: «кто существует» (durable) → `block.profile`/
`block.camp` в DB; «кто online + endpoint» (volatile) → memory из camp-announce.
Persist не нужен.

### Секрет vs публичное (внутри DB-слоя)

| данные | где | почему |
|---|---|---|
| имя, first/last, email, **публичные** passkey-креды | `block.profile` | публично, нужен синк |
| состав/админы кэмпа | `block.camp` (owner) | публичный факт допуска, single-writer |
| OIDC-клиент (id/redirect/…), публичный JWK | `block.oidcclient` / well-known | публичный конфиг |
| **client secret**, **приватный ключ IdP** | config/vault, per-peer | секрет; реплика ключа IdP сломала бы per-peer attestation |

## Создание профиля — обязательно

Профиль (first/last + passkey на `<zone>.f2f`) создаётся **в обязательном
порядке, на своём узле** — не лениво на чужом IdP.

- **Почему на своём узле:** публичную passkey-креду надо записать в `block.profile`,
  а его подписывает **только сам владелец** своим peer-ключом (ключ в helper'е
  владельца, не на чужом IdP). Полная WebAuthn-церемония на чужом IdP записать
  чужой профиль не может — это неверная кастодия ключа.
- **Когда:** при первом запуске / входе в камп (на своём узле). Так к моменту
  входа куда-либо профиль и passkey уже есть и синкнуты.
- **JIT-fallback:** если первый вход случается на чужом IdP, а профиля ещё нет,
  IdP возвращает «настрой профиль на своём узле» + deep-link на локальный UI →
  пир заводит профиль локально (helper подписывает и синкает) → повтор входа,
  креда уже в логе. Клунково, но корректно; полную церемонию на чужом IdP не
  проводим.

## Вкладка peers (UI)

Секция `peers` в боковом меню — список **по пирам** (= людям).

- **Источник:** свёртка `block.profile` (все профили) + `block.camp` (роли/членство)
  — это identity; статус — из pair-state (membership-gated) + announce (liveness,
  из memory).
- **Рендер:** строка пира (имя, бейдж owner/admin, «you», dot-статус, метка «это
  устройство»).
- **Действия:** «+ Invite» (camp-invite — только owner), revoke инвайта,
  remove peer (owner).

## Открытые вопросы

- **Bootstrap (курица-яйцо):** новичок ещё не в `block.camp.members` → синк гейтится →
  как он получит сам `block.camp`? Допуск = владелец пишет новую версию **и**
  доставляет её новичку — это ровно процесс из [INVITE.md](INVITE.md).
- **Реальное исключение (revocation):** убрать `peer_pub` из members — но старые
  фреймы и ключи у исключённого остаются. Настоящее исключение (forward-secrecy)
  требует **e2e-эпох** (смена group key при исключении), как и у каналов.
- **Делегирование:** `admins` могут менять состав? Тогда правило валидности —
  «подпись owner **или** текущего admin», и это уже не чистый single-writer
  (admin'ы могут разойтись → нужен tie-break/owner-как-арбитр). На старте — только owner.
- **Username: уникальность и ренейм.** `name` = username, должен быть уникален в
  кэмпе (email и логин строятся из него). Идентичность якорится на стабильный
  `peer_pub`; `name` — переименуемое отображаемое имя, но уникальность надо
  обеспечивать (коллизии в `profiles`).
- **Passkey sign-counter.** WebAuthn-счётчик проверяет верифицирующий IdP, а
  писать `block.profile` может только владелец → чужой IdP счётчик в профиль не
  сохранит. Либо игнорировать счётчик (многие passkey шлют 0), либо каждый IdP
  ведёт его локально (разъезжается — клон-детект слабеет, но терпимо).

## Этапность

Режем на самодостаточные куски, чтобы объём был подъёмным:

1. 📐 **Модель** — зафиксировать схемы `block.profile/camp/invite` + карту хранения
   (этот док). Кода ноль. ← *сделано здесь.*
2. 📐 **`block.profile` + вкладка peers (read-only)** — каждый пир публикует
   self-профиль; UI рисует пиров из блоков. Без инвайтов и гейтинга. Низкий риск,
   сразу видно. Заодно синк публичных passkey-кредов заменяет локальный
   `credstore.go`.
3. 📐 **`block.camp`** (owner-only членство) + подключить к `SetMemberCheck` и
   `handlePairReq` как источник членства; camp-announce → только liveness;
   выпилить `PeerCatalog` из config.
4. 📐 **Инвайты** — camp-invite (owner) + join-handshake (детали флоу/подписей —
   в [INVITE.md](INVITE.md)).
5. 📐 **`block.oidcclient`** (публичный конфиг, channel-gated); секреты → vault.
6. 📐 **e2e-эпохи** для реального исключения — общий механизм с каналами.
