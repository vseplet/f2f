# Идентичность, членство и аутентификация — на блоках

> **Статус:** 📐 дизайн. Кладёт на блок-движок (`db/blocks`) три вещи: **кто ты**
> (личность), **пускают ли тебя в кэмп** (членство, контролирует владелец), **чем
> ты входишь** (passkey/OIDC-профиль). Связано: [INVITE.md](INVITE.md) (инвайты/
> допуск), [OIDC.md](OIDC.md) (провайдер — реализован), [SECRETS.md](SECRETS.md)
> (vault), [ENTITIES.md](ENTITIES.md) (модель сущностей).

## Идея

Личность, членство и аутентификация — это всё **подписанные иммутабельные факты**,
ровно то, для чего есть лог `db`. Кладём их блоками. Одно **жёсткое правило**:

> **Публичное — в блоки. Секреты — нет.** payload блока сейчас открыт (e2e-seal —
> дизайн), поэтому любой приватный ключ в блоке утечёт во все реплики.

## Слой 1 — Личность: `block.user` (self-authored)

- **scope** `user:<user_id>`; **автор = сам пользователь** (самоутверждаемый профиль).
- **content:**
  ```jsonc
  block.user {
    user_id,                       // per-camp идентичность
    name,                          // отображаемое имя
    devices:  ["<device_pub>", …], // устройства этого юзера (ed25519)
    passkeys: [<public webauthn credential>, …]  // ТОЛЬКО публичные креды
  }
  ```
- Пользователь заявляет свои устройства и **публичные** passkey-креды. Приватный
  ключ passkey не наш — он не покидает аутентификатор, хранить нечего.
- **Зачем блоком:** passkey начинает работать на всех устройствах юзера, и любой
  co-located IdP проверяет его по публичной креде из лога. Это закрывает TODO в
  `services/oidc/credstore.go` («camp-wide replication of the public credential»).

## Слой 2 — Членство в кэмпе: `block.camp` (owner-only)

То, что раньше называли «список пиров». Это **верхний ACL кэмпа**.

- **scope** well-known `camp`; **single-writer = владелец** (pub из `camp_id`
  `<owner_pub>_<label>`).
- **Правило валидности (ключевое):** версия `block.camp` принимается **только если
  подписана владельцем**; любые чужие версии фолд отбрасывает. «Контролирует только
  владелец» получается **by construction — через подпись**, а не через проверку на
  сервере. Single-writer ⇒ ни конфликтов, ни siblings/tabs.
- **content:**
  ```jsonc
  block.camp {
    name,                          // имя кэмпа
    members: ["<user_id>", …],     // кто допущен в кэмп
    admins?:  ["<user_id>", …]     // опционально: кому делегировано право менять состав
  }
  ```
- **Член кэмпа ⟺ его `user_id` в этом блоке.** Camp-сервер больше **не решает**, кто
  внутри — он становится чистым рандеву/релеем (можно считать недоверенным).

## Слой 3 — Аутентификация / OIDC

- **`block.oidcclient`** (scope `channel:<bid>` или `camp`): публичный конфиг клиента
  ```jsonc
  block.oidcclient { client_id, name, redirect_uris, logout_uris, confidential }
  ```
  allowlist «кому можно входить» = **членство канала** (участники). Выигрывает от
  синка между устройствами/пирами и от истории версий (аудит).
- **Публичный JWK** — может быть блоком, но и так раздаётся на well-known эндпоинте.
- **НЕ в блоки (секреты):**
  - **client secret** (сейчас в `clients.json` плейнтекстом) и
  - **приватный RSA-ключ IdP** (`jwt.go`)
  → в **vault** ([SECRETS.md](SECRETS.md), device-wrapped/sealed), либо
  **per-peer-локально**. Реплика приватного ключа IdP = любой пир сможет подписывать
  токены как этот IdP → ломается per-peer attestation. **Подпись остаётся per-peer.**

## Как подключается к движку

| что | сейчас | становится |
|---|---|---|
| членство кэмпа | camp-announce (кто заанонсился) | `block.camp.members` (owner-signed) |
| `Sync.SetMemberCheck` | гейт по членству **канала** | + верхний гейт: не в составе кэмпа → ничего не реплицируем |
| `handlePairReq` (допуск пиров) | дроп не-членов по camp-роли | источник членства — `block.camp` |
| camp-announce | допуск + liveness | только **liveness** (кто сейчас online) |
| членство канала | `block.channel.members` | без изменений — **нижний слой ACL** |

Два слоя: `block.camp` (кто в кэмпе, owner) ⊃ `block.channel.members` (кто в канале,
владелец канала).

## Секрет vs публичное (правило одной таблицей)

| данные | где | почему |
|---|---|---|
| user_id, имя, device-pubs | `block.user` | публично, нужен синк |
| **публичные** passkey-креды | `block.user` | проверка passkey везде; приват в аутентификаторе |
| состав/админы кэмпа | `block.camp` (owner) | публичный факт допуска, single-writer |
| OIDC-клиент (id/redirect/…) | `block.oidcclient` | публичный конфиг, channel-gated |
| публичный JWK | блок или well-known | публичный |
| **client secret** | vault / локально | секрет |
| **приватный ключ IdP** | per-peer локально | секрет + per-peer attestation |

## Открытые вопросы

- **Bootstrap (курица-яйцо):** новичок ещё не в `block.camp.members` → синк гейтится →
  как он получит сам `block.camp`? Допуск = владелец пишет новую версию **и**
  доставляет её новичку — это ровно процесс из [INVITE.md](INVITE.md).
- **Реальное исключение (revocation):** убрать `user_id` из members — но старые
  фреймы и ключи у исключённого остаются. Настоящее исключение (forward-secrecy)
  требует **e2e-эпох** (смена group key при исключении), как и у каналов.
- **Делегирование:** `admins` могут менять состав? Тогда правило валидности —
  «подпись owner **или** текущего admin», и это уже не чистый single-writer
  (admin'ы могут разойтись → нужен tie-break/owner-как-арбитр). На старте — только owner.
- **device vs user:** членство ведём по `user_id`, устройства висят под ним
  (`block.user.devices`); привязка device→user — device-cert.
