# Инвайты и допуск пиров

> **Статус:** дизайн (не реализовано).

Описывает целевую модель приглашений и допуска. **Модель: пир = пользователь**
(идентичность = `peer_pub`, без слоя `user_id`; см. [ENTITIES.md](ENTITIES.md),
[IDENTITY.md](IDENTITY.md)). Несколько устройств = несколько пиров, каждое
заходит в кэмп своим camp-invite.

> **Реализация — на блоках.** Профиль = `block.profile`, членство кэмпа =
> `block.camp` (owner-only), инвайты-мета = `block.invite`; распространяются
> синком лога `db`. Блок-модель и **карта хранения (DB / config / memory)** — в
> [IDENTITY.md](IDENTITY.md). Этот док — про *флоу, подписи и UX*.

## Где мы сейчас (baseline)

- **Допуск открытый**: знание `camp_id` = членство. `camp_id = <owner_pub>_<label>`;
  из него же выводится обфускейт-ключ (`obfenv.NewCamp`). То есть `camp_id`
  сейчас — и адрес кэмпа, и секрет допуска одновременно.
- **`GenerateInvite`** (owner-signed bearer-токен, single-use `invite_id`, TTL)
  **определён в `identity`, но нигде не вызывается** — в join не вкручен.
  `Join` просто сохраняет camp-config по `camp_id` и генерит локальную личность.
- **Сервер rendezvous не проверяет допуск** — кто заannounce'ился с `camp_id`,
  тот в роестре.
- **Нет человеческого апрува нового пира.** `pair_req/pair_res` — это
  авто NAT-traversal + liveness (крипто-подпись), не согласие.
- **Identity = ed25519 на (машину × камп).** Это и есть пир — отдельного
  понятия пользователя нет.

Итог: «правильных» инвайтов нет — это и надо построить.

## Целевая модель

- **Членство кэмпа = множество `peer_pub`**, owner-signed в `block.camp`.
- `camp_id` перестаёт быть секретом допуска (остаётся обфускейт-ключом);
  допуск — через подписи и членство.
- Каждый пир публикует свой **профиль** (`block.profile`, self-authored) —
  имя, email, публичные passkey-креды (см. [IDENTITY.md](IDENTITY.md)).

Один артефакт допуска:

| артефакт | кто подписывает | смысл |
|---|---|---|
| **camp-invite** | owner (или делегат-admin) | впустить **новый `peer_pub`** в камп |

Каноническая строка подписи (по образцу `InviteSigMsg`):
```
camp-invite|<camp_id>|<invite_id>|<exp>
```

## Сценарий — пригласить новый пир

1. Owner выпускает **camp-invite** (owner-signed, привязан к `camp_id`,
   single-use `invite_id`, TTL). Делится ссылкой/токеном.
2. Приглашённый на своём узле: генерит `peer_pub`, создаёт **профиль** (имя,
   first/last) и **passkey** на `<zone>.f2f`, гасит токен.
3. Узел доказывает серверу/пирам: «вот camp-invite, подписанный owner'ом» →
   `peer_pub` добавляется в `block.camp.members`, профиль синкается всем.

Owner уже авторизовал (подписал инвайт), поэтому отдельный апрув не обязателен
— но опционально можно (trust-on-first-contact, ниже).

> **Второе устройство того же человека** = просто ещё один camp-invite на новый
> `peer_pub`. Отдельного «device-enroll» нет — это другой пир со своим профилем
> и своим passkey. Так решили «для простоты».

## Апрув соединения с новым пиром

Сейчас отсутствует. Опционально добавляем **trust-on-first-contact** гейт:

- При первой встрече нового `peer_pub` (попытка pairing / появление в роестре)
  владельцу показывается запрос: «`<peer>` хочет подключиться — разрешить?».
- До апрува pairing/трафик к нему не поднимается (или ограничен).
- Решение запоминается (список известных/одобренных пиров).
- Для приглашённых по camp-invite апрув можно авто-проставить (owner подписал
  инвайт), чтобы не дёргать дважды.

## Что меняется в коде (наметки)

- **`identity`**: passkey-anchor профиля, инвайт-тип `camp-invite`, функции
  выпуска/проверки.
- **Join-флоу (`cli`, web)**: закрыть open-by-camp_id; гасить токен; провести
  создание профиля + passkey-энролл на узле приглашённого.
- **`rendezvous`/roster**: членство по `peer_pub` (источник — `block.camp`),
  проверка подписей сервером и/или пирами.
- **`engine` pairing**: гейт «approve new peer» + хранилище одобренных пиров.
- **OIDC** (`services/oidc`): `sub` = `peer_pub` (из attested `X-F2F-Peer`),
  email/имя из `block.profile` — см. [OIDC.md](OIDC.md).
- **UI**: профиль (своё имя/passkey), кнопка «пригласить пир», диалог апрува
  нового пира.

## Отзыв (revocation)

Нужен с самого начала, иначе «пригласил» необратимо:

- **peer revoke** (выгнать из кэмпа / потеряли машину) — owner убирает `peer_pub`
  из `block.camp.members` (новая owner-signed версия).
- Распространяется тем же синком лога, что и членство.
- Настоящее исключение (forward-secrecy: старые ключи/фреймы у исключённого
  остаются) требует **e2e-эпох** — см. [IDENTITY.md](IDENTITY.md) / [ROADMAP.md](ROADMAP.md).

## Открытые вопросы

- **Делегирование**: кто кроме owner'а может выпускать camp-invite (роль
  `@admins`, см. политики в ARCHITECTURE.md).
- **Переходный период**: пока часть пиров на open-by-camp_id — как сосуществуют.
- **Username-коллизии**: `name` в профиле уникален в кэмпе; как разрешать гонку
  двух новичков на одно имя.

## Поэтапно

1. **Фундамент**: членство по `peer_pub` в `block.camp`, профиль + passkey-якорь,
   OIDC `sub = peer_pub`. Закрыть open-join, вкрутить owner **camp-invite**.
2. **approve-new-peer** гейт + **revocation**.
3. Делегирование/роли.
