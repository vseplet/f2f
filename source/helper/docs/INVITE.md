# Инвайты, пользователи и допуск пиров

Дизайн (не реализован). Описывает целевую модель приглашений и допуска,
построенную поверх **user-identity** (см. [OIDC.md](OIDC.md), вариант C:
`user_id`, заякоренный на passkey-аккаунте; `device_pub` — ключ устройства,
привязанный к `user_id`).

## Где мы сейчас (baseline)

- **Допуск открытый**: знание `camp_id` = членство. `camp_id = <owner_pub>_<label>`;
  из него же выводится обфускейт-ключ (`obfenv.NewCamp`). То есть `camp_id`
  сейчас — и адрес кэмпа, и секрет допуска одновременно.
- **`GenerateInvite`** (owner-signed bearer-токен, single-use `invite_id`, TTL)
  **определён в `identity`, но нигде не вызывается** — в join не вкручен.
  `Join` просто сохраняет camp-config по `camp_id` и генерит локальную личность.
- **Сервер rendezvous не проверяет допуск** — кто заannounce'ился с `camp_id`,
  тот в роестре.
- **Нет человеческого апрува нового пира.** `pair_req/pair_res` — это
  авто NAT-traversal + liveness (крипто-подпись), не согласие.
- **Identity = ed25519 на (машину × камп).** Понятия пользователя нет.

Итог: «правильных» инвайтов нет — это и надо построить.

## Целевая модель

- **Членство кэмпа = множество `user_id`** (людей), а не device-pub.
- Каждый `user_id` объявляет свои **устройства** (`device_pub`), каждое —
  с **device-cert** (подпись user-ключа: «это устройство моё»).
- `camp_id` перестаёт быть секретом допуска (остаётся обфускейт-ключом);
  допуск — через подписи и членство.

Три ключа/подписи:

| артефакт | кто подписывает | смысл |
|---|---|---|
| **user-invite** | owner (или делегат) | впустить **новый `user_id`** в камп |
| **device-enroll** | сам user | привязать **новое устройство** к своему `user_id` |
| **device-cert** | сам user | постоянная привязка `device_pub → user_id` в роестре |

Канонические строки подписи (по образцу `InviteSigMsg`):
```
user-invite|<camp_id>|<invite_id>|<exp>
device-enroll|<camp_id>|<user_id>|<device_pub>|<exp>
device-cert|<user_id>|<device_pub>
```

## Сценарий 1 — пригласить нового пользователя (+ его первое устройство)

1. Owner выпускает **user-invite** (owner-signed, привязан к `camp_id`,
   single-use, TTL). Делится ссылкой/токеном.
2. Приглашённый на первом устройстве: генерит `device_pub`, создаёт **passkey**
   (это фиксирует его `user_id`), гасит токен.
3. Узел доказывает серверу/пирам: «вот user-invite, подписанный owner'ом» →
   `user_id` добавляется в членство, первое устройство привязано (device-cert).

Owner уже авторизовал (подписал инвайт), поэтому отдельный апрув не обязателен
— но опционально можно (§3, trust-on-first-contact).

## Сценарий 2 — добавить собственное устройство (та же личность)

1. Существующий user на устройстве A выпускает **device-enroll** токен —
   подписан **его user-ключом/passkey** (НЕ owner'ом), привязан к его `user_id`
   + `camp_id`, короткий TTL.
2. Новое устройство B: генерит свой `device_pub`, гасит токен → получает
   device-cert и привязывается к тому же `user_id`.
3. **Новых членов кэмпа не добавляется** — просто +устройство под существующим
   пользователем. B валиден, потому что `user_id` уже член, а его user-подпись
   доказывает «B — моё».

Частный случай: если **passkey синкается** (iCloud Keychain / Google), на B он
уже есть → B аутентифицируется тем же аккаунтом, и явный enroll-токен может не
понадобиться (passkey-ассерция сама доказывает принадлежность).

## Сценарий 3 — апрув соединения с новым пиром

Сейчас отсутствует. Добавляем **trust-on-first-contact** гейт:

- При первой встрече нового `device_pub` (попытка pairing / появление в
  роестре) владельцу показывается запрос: «`<user>/<device>` хочет подключиться
  — разрешить?».
- До апрува pairing/трафик к нему не поднимается (или ограничен).
- Решение запоминается (список известных/одобренных устройств).
- **Уровень апрува — на пользователя, а не на устройство** (рекомендация):
  одобрил человека → его новые устройства проходят молча, потому что их вучит
  его user-ключ (device-cert). Иначе пришлось бы апрувить каждый ноут.
- Для §1/§2 апрув можно авто-проставить (owner подписал инвайт / user подписал
  enroll), чтобы не дёргать дважды.

## Что меняется в коде (наметки)

- **`identity`**: user-key/passkey-anchor, device-cert, расширить инвайт-типы
  (`InviteBody` + новый `DeviceEnrollBody`), функции выпуска/проверки.
- **Join-флоу (`cli`, web)**: закрыть open-by-camp_id; гасить токен; провести
  passkey-энролл на первом устройстве.
- **`rendezvous`/roster**: членство по `user_id`, объявление device-cert,
  проверка подписей сервером и/или пирами; группировка устройств по user.
- **`engine` pairing**: гейт «approve new peer» + хранилище одобренных
  устройств; интеграция с device-cert (доверенный user → молча).
- **OIDC** (`services/oidc`): `sub` = `user_id` (резолв attested `device_pub →
  user_id`), passkey `userHandle = user_id` — см. OIDC.md.
- **UI**: профиль (мои устройства), кнопки «пригласить пользователя» /
  «добавить своё устройство», диалог апрува нового пира.

## Отзыв (revocation)

Нужен с самого начала, иначе «пригласил» необратимо:

- **device revoke** (потерял ноут) — user помечает свой `device_pub`
  отозванным; его device-cert перестаёт приниматься.
- **user revoke** (выгнать из кэмпа) — owner убирает `user_id` из членства.
- Хранится в членстве/роестре (owner-signed для user-revoke, user-signed для
  device-revoke), распространяется тем же каналом, что и членство.

## Открытые вопросы

- Где живёт **user-key** между устройствами, если не полагаться только на синк
  passkey (резерв для security-key / Linux / кросс-экосистема).
- **Делегирование**: кто кроме owner'а может выпускать user-invite (роль
  `@admins`, см. политики в ARCHITECTURE.md).
- **Переходный период**: пока часть пиров на open-by-camp_id — как сосуществуют.
- **Распространение членства/cert'ов**: через owner-mediated camp-config или
  membership-лог из [MESSAGING_DESIGN.md](MESSAGING_DESIGN.md).

## Поэтапно

1. **Фундамент**: `user_id` + `device_pub → user_id`, passkey-якорь, OIDC
   `sub = user_id`. Закрыть open-join, вкрутить owner **user-invite** (§1).
2. **device-enroll** токен (§2) — добавить своё второе устройство.
3. **approve-new-peer** гейт (§3) + **revocation**.
4. Делегирование/роли; messenger user-native (адресация/membership по `user_id`).
