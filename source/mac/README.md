# f2f-mac

macOS-сторона проекта f2f. CLI-утилита, которая поднимает виртуальный сетевой
интерфейс `utunN` и заворачивает в него трафик к указанным IP-адресам и
доменам. На текущей итерации программа **только логирует** перехваченные
пакеты — никакого пира, hole punching и forwarding ещё нет. Это «насос», к
которому потом приклеится остальная логика туннеля.

## Зависимости

- macOS (проверено на Apple Silicon)
- Go 1.22+
- Права `sudo` для открытия utun и правки таблицы маршрутизации

## Сборка

```sh
go build -o f2f-mac ./cmd/f2f-mac
```

## Запуск

```sh
sudo ./f2f-mac run --intercept 1.1.1.1,example.com
```

Опции:

- `--intercept` — список через запятую: IP, CIDR, или домен. Домены резолвятся
  один раз при старте, для каждого полученного адреса добавляется host-маршрут
  через `utunN`.
- `--local-ip` (по умолчанию `10.99.0.1`) — локальный конец p2p-адреса на
  `utunN`.
- `--peer-ip` (по умолчанию `10.99.0.2`) — удалённый конец p2p-адреса. На этом
  этапе чисто декоративный — нужен только чтобы корректно поднять интерфейс.

Ctrl+C аккуратно сворачивает работу: удаляет добавленные маршруты, закрывает
интерфейс. Интерфейс `utunN` существует ровно пока жив процесс.

## Проверка работы

В одном терминале запускаем:

```sh
sudo ./f2f-mac run --intercept 1.1.1.1
```

В выводе должно быть что-то вроде:

```
15:24:00.123456 opened utun5 (local=10.99.0.1 peer=10.99.0.2 mtu=1420)
15:24:00.124000 route 1.1.1.1/32 → utun5
```

В другом терминале:

```sh
ping -c 3 1.1.1.1
```

Пинги **не дойдут** (мы пакеты дропаем), но в логе `f2f-mac` каждые секунду
должна появляться строка:

```
15:24:01.555000 [utun5] IPv4 10.99.0.1 → 1.1.1.1 ICMP len=84
```

Это значит — перехват работает: ОС вместо отправки пакетов в физическую сеть
заворачивает их в наш userspace-процесс. То, ради чего всё и затевалось.

Для проверки чтения сырых пакетов снаружи (debug-канал) удобно держать рядом:

```sh
sudo tcpdump -i utun5 -n
```

## Известные ограничения этого этапа

- Перехватываемые домены резолвятся **один раз при старте**. Если у домена
  ротация IP (CDN, балансировщики), часть запросов пойдёт мимо. Будет
  решаться позже — либо split-DNS, либо fake-IP пулом.
- Нет IPv6-маршрутов через DHCPv6/SLAAC — IPv6 поддерживается на уровне парсера
  пакетов и `route -inet6`, но автоконфигурации интерфейса под v6 пока нет.
- Программа дропает пакеты после логирования. Для TCP это значит, что соединения
  не устанавливаются — это ожидаемое поведение на текущем шаге.
