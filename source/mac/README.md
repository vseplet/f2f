# f2f-mac

macOS-сторона проекта f2f. CLI-утилита, которая поднимает виртуальный сетевой
интерфейс `utunN`, заворачивает в него трафик к указанным IP-адресам/доменам,
и (опционально) гоняет эти пакеты в UDP-канал с другим экземпляром себя.
Туннеля «в интернет» через второй конец пока нет — он появится, когда
прикрутим userspace TCP/IP-стек на egress-стороне.

## Зависимости

- macOS (проверено на Apple Silicon)
- Go 1.22+
- `sudo` для подкоманды `run` (нужен открытию utun и правке таблицы маршрутизации).
  Подкоманде `echo` sudo не нужен.

## Сборка

```sh
go build -o f2f-mac ./cmd/f2f-mac
```

## Подкоманды

### `run`

Поднимает `utun`, добавляет host-маршруты для `--intercept`, и:

- если задана пара `--listen` + `--peer` — гоняет пакеты `utun ↔ UDP-пир` (peer mode);
- если задан `--echo-icmp` без пира — отвечает на ICMP Echo Requests локально;
- иначе — просто логирует и дропает.

Флаги:

| флаг | описание |
| --- | --- |
| `--intercept` | список через запятую: IP, CIDR, домен. Домены резолвятся один раз на старте. |
| `--local-ip` / `--peer-ip` | точка-точка адреса на `utun` (по умолчанию `10.99.0.1`/`10.99.0.2`). |
| `--listen` | UDP-адрес для приёма от пира, например `:9000`. |
| `--peer` | UDP-адрес пира, например `127.0.0.1:9001`. |
| `--echo-icmp` | без пира: отвечать на ICMP Echo Request локально. |

Ctrl+C — чистое завершение: маршруты удаляются, `utun` закрывается.

### `echo`

UDP-only утилита для теста: принимает пакет, если это IPv4 ICMP Echo Request —
переписывает в Echo Reply и отправляет обратно отправителю. Без `utun`, без `sudo`.
Это «дальний конец» туннеля для шага A.

```sh
f2f-mac echo --listen :9001
```

## Сценарии проверки

### Single-process (без UDP-транспорта)

Базовый перехват с локальным ICMP-echo:

```sh
sudo ./f2f-mac run --intercept 1.1.1.1 --echo-icmp
ping -c 3 1.1.1.1     # 0% loss, RTT доли миллисекунды — всё в памяти
```

В логе действие — `[echo]`.

### Two-process (UDP-транспорт между двумя экземплярами)

Цель — доказать, что трубопровод «`utun_A` → UDP → echo-сервер → UDP → `utun_A`»
работает. Используем `198.51.100.5` (TEST-NET-2, RFC 5737 — гарантированно не
существует в реальном интернете, никаких коллизий с другими маршрутами).

Терминал 1 (`run`, нужен sudo):

```sh
sudo ./f2f-mac run \
  --intercept 198.51.100.5 \
  --listen :9000 --peer 127.0.0.1:9001
```

Терминал 2 (`echo`, sudo не нужен):

```sh
./f2f-mac echo --listen :9001
```

Терминал 3:

```sh
ping -c 3 198.51.100.5
```

Ожидаемое:

- `ping` показывает `0% packet loss`, RTT — около миллисекунды (loopback).
- В терминале 1: чередуются строки `[utun5] … [→peer]` (туда) и `[udp 127.0.0.1:9001] … [→utun]` (обратно).
- В терминале 2: каждые секунду строка `[udp …] IPv4 10.99.0.1 → 198.51.100.5 ICMP … [echo]`.

Если ping молчит — самое полезное место для отладки:

```sh
sudo tcpdump -i utun5 -n
netstat -rn | grep utun
```

## Известные ограничения этого этапа

- Перехватываемые домены резолвятся **один раз при старте**. Если у домена
  ротация IP (CDN, балансировщики), часть запросов пойдёт мимо. Будет
  решаться позже — либо split-DNS, либо fake-IP пулом.
- IPv6 на уровне парсера и `route -inet6` поддерживается, но автоконфигурации
  адресов под v6 на `utun` пока нет.
- UDP-транспорт сейчас передаёт **сырые IP-пакеты 1:1** — никакого framing,
  никаких stream-ID. Это сознательно — на шаге А нужен только базовый пайп.
