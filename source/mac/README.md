# f2f-mac

macOS-сторона проекта f2f. CLI-утилита, которая поднимает виртуальный сетевой
интерфейс `utunN`, опционально заворачивает в него трафик к указанным IP/доменам
и гоняет пакеты в UDP-канал со вторым экземпляром себя. Тот же бинарь играет
оба конца туннеля — отличия только во флагах.

На данном этапе это «голый L3-туннель» — два `utun` соединены через UDP,
никакого NAT и выхода в реальный интернет ещё нет. Это появится дальше.

## Зависимости

- macOS (проверено на Apple Silicon)
- Go 1.22+
- `sudo` для открытия `utun` и правки таблицы маршрутизации

## Сборка

```sh
go build -o f2f-mac ./cmd/f2f-mac
```

## Запуск

`f2f-mac run` — единственная команда. Флаги:

| флаг | описание |
| --- | --- |
| `--intercept` | список через запятую: IP, CIDR, домен. Маршруты `/32` через `utun`. Необязателен. |
| `--local-ip` / `--peer-ip` | точка-точка адреса на `utun` (по умолчанию `10.99.0.1`/`10.99.0.2`). |
| `--listen` | UDP-адрес для приёма от пира, например `:9000`. |
| `--peer` | UDP-адрес пира, например `10.0.0.5:9000`. |

`--listen` и `--peer` или оба заданы (peer mode), или оба пустые (тогда просто
открываем `utun` и логируем пакеты, никуда не пересылая).

Ctrl+C — чистое завершение: маршруты удаляются, `utun` закрывается.

## Сценарии

### Локальный single-process (без UDP-транспорта)

Базовый перехват: убеждаемся что роуты работают и пакеты идут в `utun`.

```sh
sudo ./f2f-mac run --intercept 198.51.100.5
ping -c 3 198.51.100.5
```

`ping` молчит (мы пакеты дропаем), но в логе видны строки
`[utun5] IPv4 10.99.0.1 → 198.51.100.5 ICMP len=84 [drop]`.

### Симметричный two-process (полный туннель между двумя экземплярами)

Каждый экземпляр поднимает свой `utun` и гоняет всё, что в него прилетело,
по UDP в пир. Адреса `--local-ip`/`--peer-ip` зеркалятся:

```sh
# A (например, ingress — заворачивает что-то в туннель):
sudo ./f2f-mac run \
  --intercept 198.51.100.5 \
  --local-ip 10.99.0.1 --peer-ip 10.99.0.2 \
  --listen :9000 --peer 10.0.0.5:9000

# B (egress конец):
sudo ./f2f-mac run \
  --local-ip 10.99.0.2 --peer-ip 10.99.0.1 \
  --listen :9000 --peer A_PUBLIC_IP:9000
```

После запуска двух экземпляров (на разных машинах!) `ping 10.99.0.2` с A
ходит сквозь туннель, ОС B сама отвечает на ICMP, ответ возвращается обратно.
Это работает «как настоящая локалка между двумя машинами».

⚠️ **На одной машине это симметрично не запустить:** оба процесса будут
бороться за одни и те же IP в один routing table, конфликты неизбежны.
Для тестов нужны либо две физические машины, либо Linux VM (Lima) как
второй хост.

## Известные ограничения этого этапа

- Маршрут только для адресов, явно указанных в `--intercept`. «Заворачивать
  всё подряд» (default route) пока не делаем — это специально, чтобы наш UDP к
  пиру случайно не попал в собственный туннель.
- Домены резолвятся **один раз при старте**. Если у домена ротация IP (CDN) —
  часть запросов пойдёт мимо. Будет решаться позже: split-DNS или fake-IP пул.
- Нет NAT/forwarding на egress-стороне. То есть пинг между `utun`-IP двух
  машин работает, но «выйти через B в интернет» — ещё нет. Это следующий шаг.

## Полезные команды для отладки

```sh
# Что у нас в утунах и маршрутах
ifconfig | grep -A2 utun
netstat -rn | grep utun

# Снифать сами utun-пакеты:
sudo tcpdump -i utun5 -n -vv

# Снифать UDP-трафик между экземплярами:
sudo tcpdump -i lo0 -n udp port 9000        # локально
sudo tcpdump -i en0 -n udp port 9000        # по сети
```

Wireshark (`brew install --cask wireshark`) видит все интерфейсы, включая
`utun*` и `lo0`, и сильно нагляднее для разбора потоков.
